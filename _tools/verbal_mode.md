# Tool modifier: VERBAL MODE (spoken answers)

When Soham dictates answers (via macOS dictation or the local `dictate` script), his messages are
**transcribed speech**, not typed prose. Grade and treat them accordingly. This applies on top of
any command (`drill`, `mock`, `case`, etc.).

## How to read a transcribed answer
- **Ignore transcription artifacts:** missing punctuation, run-on sentences, homophones (e.g. "loss"/"laws"), mis-rendered jargon ("NDCG" → "n d c g", "logits" → "logits"/"low jits"). Interpret charitably — grade the *idea*, not the transcription.
- If a term is garbled but the meaning is clear from context, assume Soham said it correctly.
- If a garble makes the answer ambiguous, ask a quick clarifying question rather than penalizing.

## Add a "spoken delivery" dimension to grading (verbal mode only)
On top of the standard rubric, score how this would land **said aloud to an interviewer**:
- **Conciseness:** did it ramble or stay tight? Spoken answers must be ~60–120s of content for a concept.
- **Structure aloud:** did he signpost ("three reasons: first… second…")? Verbal answers live or die on structure.
- **Filler / hedging:** flag excessive "um, like, I think, kind of, not sure" — these read as low confidence in interviews.
- **Lead with the answer:** did he state the conclusion first, then justify? (Strong) Or wander toward it? (Weak)

## Feedback addition (append to normal rubric feedback)
```
🎙️ Spoken delivery: <tight / rambling / well-structured / buried the lead / too much hedging>
   → one concrete fix for saying it better aloud.
```
